# 反模式手册（真实项目踩坑经验）

以下所有反模式都是在真实项目中验证过的"导致相似度低或返工"的做法。

## 1. 一个 Agent 搞定所有屏幕

**症状**：看起来节省了 Agent 调用次数
**后果**：Agent 上下文爆炸、决策质量下降、最终 17-30% 相似度
**正解**：按模块切分，每 Agent 负责 1-8 个屏幕

## 2. 凭截图目测写代码

**症状**：Agent 说"看起来是紫色"、"大约 16pt"
**后果**：永远差一点点，Phase 4 发现大量偏差
**正解**：强制用 `batch_get` 取 JSON，精确字段必须匹配

## 3. 跳过 Phase 2 直接写代码

**症状**：觉得"规格文档多余"
**后果**：结构错误 (比如缺少外层卡片)，Phase 4 审计时才发现，大量返工
**正解**：spec → code，顺序严格

## 4. 同一个 Agent 既写又审

**症状**：为了省事，Phase 3 Agent 直接汇报"我这屏 95%"
**后果**：盲点，实际可能 70%
**正解**：Phase 4 必须是独立的 Agent 组

## 5. 弹窗留到最后做

**症状**：先把 A 屏幕写完，发现需要弹窗 → 造个弹窗1 → B 屏幕又要弹窗 → 造个弹窗2...
**后果**：5 个不同风格的弹窗并存，最后得统一重写
**正解**：Phase 1 把所有弹窗类型一次性建完

## 6. 硬编码颜色/字号

**症状**：
```objc
label.textColor = [UIColor colorWithRed:0.5 green:0.3 blue:0.8 alpha:1];
```
**后果**：设计 Token 更新时要全局查找替换，易漏
**正解**：所有数值走 Phase 1 建立的常量

## 7. 跨模块修改文件

**症状**：Agent A 负责 Home，但顺手改了 Common/Utils
**后果**：和 Agent B 冲突、破坏其他模块
**正解**：Agent 只改自己模块，通用组件改动由指挥官协调

## 8. 编译都不跑就说"完成"

**症状**：Agent 最后报告"所有文件已创建"
**后果**：用户一编译崩溃
**正解**：每 Phase 结束必须 build，最后必须在 simulator 运行

## 9. 把重复出现的结构当独立个案

**症状**：CreditInfo 3 种状态（审核/待还款/待提取）写成 3 个 Controller
**后果**：一个状态改了，另外两个忘了，三套代码漂移
**正解**：Phase 0 识别 variant group，Phase 3 用一个 Controller + 状态切换

## 10. 图片资源留到最后

**症状**：所有 UIImageView 都用 SF Symbol 占位
**后果**：最后导图片时要改 50 处代码
**正解**：Phase 0 就导出所有图片，Phase 3 写代码时直接引用

## 11. 不做跨屏一致性检查

**症状**：导航栏在 Home 是透明的，在 Settings 是不透明的
**后果**：用户一眼看出风格不统一
**正解**：指挥官在 Phase 2 结束时必须 review 所有 spec

## 12. 忽略空状态

**症状**：列表空的时候显示空白屏
**后果**：设计图有空状态，用户反馈"没做"
**正解**：Phase 0 识别空状态、加载态、错误态；Phase 1 建 SCEmptyStateView

## 13. Phase 5 越修越乱

**症状**：修一个 bug 引入两个新 bug
**后果**：越修相似度越低
**正解**：
- 最小改动原则
- 每改完一个立刻验证
- Phase 5 Agent 之间不共享文件

## 14. 不导 Pencil JSON 就开始分析

**症状**：直接问"这屏大概长什么样"
**后果**：数据漂移、版本不一致
**正解**：每个 Phase 开始都重新 `batch_get` 拿最新数据

## 15. 用 Skill 不看 Skill

**症状**：用户调 `/design-to-code` 但 Agent 没读 `SKILL.md` 就开干
**后果**：跳过 Phase 0，直接写屏幕，回到老问题
**正解**：Skill 开头明确要求"先读 SKILL.md 再读 design_intel.md 再开工"
